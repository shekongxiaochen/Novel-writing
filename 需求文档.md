## 1. 背景与目标

### 1.1 背景
小说作者在创作过程中会产生大量零散记录（大纲、设定、伏笔、人物关系等）。本产品旨在将这些信息**结构化**，帮助作者理清叙事脉络、减少前后矛盾，并在日常写作中提供接近常见写小说软件的基础体验。

### 1.2 产品目标
1. **写作**：支持按作品组织章节与正文，满足日常码字与导航的基础需求。
2. **脉络**：支持大纲与关键结构信息的管理，便于把握全书走向。
3. **设定与人物**：支持角色、势力、世界观等档案的维护与检索。
4. **质量跟踪**：支持伏笔、坑点、矛盾与待办的可追踪记录。

### 1.3 非目标（当前概要版不展开）
- 具体技术栈、数据库、接口定义（见独立技术文档）。
- 思维导图、地图等可视化形态的实现细节（见功能总表「预留」项）。
- AI 辅助写作、多作者协作的详细规则（第三期及以后单独定义）。

## 2. 范围与分期

范围以《功能总表.md》**第八章**为准，概要如下：

| 阶段 | 范围摘要 |
|------|----------|
| 第一期 | 作品与元数据、章/卷与正文写作、大纲、角色与势力档案、伏笔/坑点 |
| 第二期 | 故事线/时间线、角色关系网与势力关系网（可先表格化再可视化）、设定库与世界观强化 |
| 第三期 | 思维导图、地图等可视化、AI 辅助、多作者协作 |

**本需求文档**按《功能总表》**第一至七章**逐块列出需求条目；分期仅标注优先级建议，具体排期以项目计划为准。

## 3. 用户与典型场景

### 3.1 目标用户
以长篇/连载小说作者为主，需要同时管理「正文」与「设定、大纲、伏笔」的作者。

### 3.2 典型场景
1. 新建作品，填写书名与基本元数据，进入该作品的写作空间。
2. 建立卷/章结构，在某一章中撰写正文，并查看字数与章级备忘。
3. 维护全书或分卷大纲，将情节要点分层级记录。
4. 建立角色与势力档案，补充属性与关系（第二期起强化关系网）。
5. 记录伏笔与坑点，标记是否回收或待处理，避免遗漏。

## 4. 功能需求（与《功能总表》对应）

以下编号 **FR-x.y** 中，**x** 与《功能总表》章节序号一致，便于追溯。

---

### 4.1 作品与写作 — 对应功能总表「一」

| 编号 | 需求描述 | 优先级建议 |
|------|----------|------------|
| FR-1.1 | 用户可创建新作品（小说项目），创建后可进入该作品。 | P0 |
| FR-1.2 | 用户可为作品维护元数据：至少包括书名、简介；可扩展题材/标签、叙事视角与基调、是否多线叙事等约定、作者备忘。 | P0 |
| FR-1.3 | 用户可在作品内组织卷、章（及若需要的节），支持排序、重命名等基础结构操作。 | P0 |
| FR-1.4 | 用户可在作品内进行正文写作，具备常见写小说软件应具备的基础编辑与阅读体验（具体能力在详细需求中列举）。 | P0 |
| FR-1.5 | 用户可为单章维护笔记或备忘：如本章写作提示、待改点、与大纲的对应说明。 | P1 |
| FR-1.6 | 用户可在作品内检索或导航至目标章节，例如按标题或关键词跳转、侧边章节目录。 | P1 |
| FR-1.7 | 系统可提供字数统计；可选支持篇幅或阶段目标（概念级，非强制）。 | P1 |
| FR-1.8 | 用户可将正文及设定等内容导出或备份；具体格式在详细需求中约定。 | P2 |

---

### 4.2 脉络与结构 — 对应功能总表「二」

| 编号 | 需求描述 | 优先级建议 |
|------|----------|------------|
| FR-2.1 | 用户可维护大纲：支持全书/卷/章等层级的情节要点与层级结构。 | P0（第一期） |
| FR-2.2 | 用户可维护故事线或时间线：表达多线叙事、事件先后顺序、因果与并行关系（第二期细化）。 | P1 |
| FR-2.3 | 用户可记录冲突与主线目标、阶段目标、关键转折点；宜与大纲或故事线建立概念关联。 | P1 |
| FR-2.4 | （可选）用户可标注节奏或预计篇幅等，用于控制叙事节奏。 | P2 |
| FR-2.5 | 产品预留思维导图能力：用于以导图形式整理情节或主题分支，实现形态第三期定义。 | 预留 |
| FR-2.6 | 产品预留故事线可视化能力：与故事线/时间线数据同源，第三期定义。 | 预留 |

---

### 4.3 地理与世界观 — 对应功能总表「三」

| 编号 | 需求描述 | 优先级建议 |
|------|----------|------------|
| FR-3.1 | 用户可维护世界观总览：如时代背景、规则体系、社会结构等总述。 | P1（第二期强化） |
| FR-3.2 | 用户可维护设定库：条目可分类（如地点、物品、规则、名词表等），并支持检索。 | P1 |
| FR-3.3 | 产品预留地图能力：地理关系、路线、势力分布等，第三期展开。 | 预留 |
| FR-3.4 | 设定宜能与章节、大纲或情节节点建立引用或关联关系，减少设定与正文脱节（详细规则第二期定）。 | P2 |

---

### 4.4 角色与势力 — 对应功能总表「四」

| 编号 | 需求描述 | 优先级建议 |
|------|----------|------------|
| FR-4.1 | 用户可为作品维护角色档案：包括但不限于姓名、性别、年龄、外貌、性格、目标、秘密、成长弧等字段（字段集可在详细需求中扩展）。 | P0（第一期） |
| FR-4.2 | 用户可维护角色之间的关系：关系类型及随剧情变化（第二期可与关系网视图结合）。 | P1 |
| FR-4.3 | 用户可维护势力/组织档案：名称、目标、资源、领袖、与主角关系等。 | P0（第一期） |
| FR-4.4 | 用户可维护势力间关系：如联盟、敌对、从属、隐藏关系等。 | P1 |
| FR-4.5 | （可选）记录角色在章节或情节中的出场，便于查漏。 | P2 |

---

### 4.5 质量、伏笔与待办 — 对应功能总表「五」

| 编号 | 需求描述 | 优先级建议 |
|------|----------|------------|
| FR-5.1 | 用户可记录伏笔：埋设位置、计划回收位置、是否已回收等状态。 | P0（第一期） |
| FR-5.2 | 用户可维护坑点/疑问清单：如逻辑漏洞、未解释点、时间线疑问、人设是否 OOC 等，并具备分类或标签能力（详细需求中定义枚举）。 | P0 |
| FR-5.3 | 用户可维护矛盾清单：已发现的设定冲突、待决策点。 | P1 |
| FR-5.4 | 用户可维护写作待办：如本章任务、待补设定、待改前文。 | P1 |

## 5. 非功能需求（概要）

| 编号 | 类别 | 描述 |
|------|------|------|
| NFR-1 | 数据与安全 | 用户数据需有备份与恢复预期；敏感信息处理遵循后续安全规范。 |
| NFR-2 | 性能 | 单作品常规规模下编辑与导航响应可接受；具体指标在详细需求中量化。 |
| NFR-3 | 可用性 | 核心写作与大纲路径清晰，减少作者切换心智成本。 |
| NFR-4 | 扩展性 | 预留与故事线、地图、导图、AI 的数据结构衔接空间，避免推倒重来。 |

---

## 6. 假设与依赖

1. 第一期以「单用户 + 完整创作闭环（作品—章—正文—大纲—角色/势力—伏笔）」为验收主线。
2. 可视化类功能（导图、地图）以《功能总表》预留项为准，不在第一期强制交付。
3. 与《功能总表》不一致时，以经评审后的《功能总表》更新为准。
