# AI 写作能力测试系统 — 设计方案

> 目的：把"改了提示词/参数/模型，AI 到底有没有变强"从**凭感觉**变成**有数据**。
> 定位：**离线开发工具**，只给作者自己调优时用，不接入用户端、不随客户端分发。

---

## 一、设计目标与原则

### 核心目标
每次改动（提示词、采样参数、模型、上下文工程）后，跑一次测试，得到一张可对比的成绩单，明确回答："这次改动让 AI 变强了还是变弱了，强在哪、弱在哪。"

### 四条地基原则（决定一切设计）
1. **固定测试集**：输入永远不变、版本化存仓库。输入变了，结果就不可比。
2. **多次重复取中位数**：每个场景跑 3~5 次，抵消生成随机性。单次结果无意义。
3. **一次只改一个变量**：固定其余条件，只换被测项，才能归因。
4. **盲测 A/B 优先于绝对打分**：裁判同时看两版输出（顺序随机），判谁更好；比孤立打分可靠得多。

### 不做什么（避免跑偏）
- 不追求"绝对分数权威性"，只追求"版本间相对涨跌可信"。
- 不接入用户界面，不消耗用户余额（用作者自己的 key）。
- 不一次做全：先做零成本的自动指标，再做花钱的 AI 裁判。

## 二、整体架构

```
┌─────────────────────────────────────────────────────────────┐
│  测试集 (TestSuite)          固定输入，版本化存仓库              │
│  ├─ 场景1: 开篇             scenario.json                      │
│  ├─ 场景2: 战斗动作                                            │
│  ├─ 场景3: 对话交锋                                            │
│  ├─ 场景4: 情感戏                                              │
│  ├─ 场景5: 章尾钩子                                            │
│  └─ 场景6: 长上下文续写(测一致性)                              │
└───────────────────────────┬─────────────────────────────────┘
                            │  ① 按"被测配置"逐场景生成 N 次
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  被测配置 (RunConfig)                                          │
│  { model, temperature, penalties, promptVersion, label }      │
└───────────────────────────┬─────────────────────────────────┘
                            │  ② 每份输出过一组打分器
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  打分器 (Scorers)                                              │
│  A层 自动指标(代码算, 免费)   B层 AI裁判(强模型, 花钱)          │
│  · AI腔密度  · 重复度          · 文笔  · 创意  · 一致性          │
│  · 套话句式  · 指令遵循        · 角色声音  · 钩子吸引力          │
└───────────────────────────┬─────────────────────────────────┘
                            │  ③ 汇总成一张成绩单
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  成绩单 (Report)  存历史 → 与上一版并排对比 → 可视化界面         │
└─────────────────────────────────────────────────────────────┘
```

### 数据流核心概念
- **Scenario（场景）**：一个固定的测试输入（含上下文、指令、可选的"已知事实"用于查一致性）。
- **RunConfig（被测配置）**：一次要测的完整配置组合，带一个人类可读 `label`（如 `temp0.8+新提示词v3`）。
- **Run（一次跑测）**：用某个 RunConfig 跑完整个测试集（每场景 N 次），产出一批输出。
- **Report（成绩单）**：对一次 Run 的所有输出打分后的汇总，版本化存档。
- **Comparison（对比）**：两张 Report 并排，逐维度看涨跌。

## 三、测试集设计（Scenario）

### 场景分类（覆盖能力的各个面）
| # | 场景类型 | 测什么能力 | 输入要点 |
|---|---|---|---|
| 1 | 开篇冷启动 | 抓人能力、不落俗套开头 | 仅给设定+基调，要求写第一章前 800 字 |
| 2 | 战斗/动作 | 节奏、画面感、不注水 | 给对阵双方+环境，要求一场打斗 |
| 3 | 对话交锋 | 潜台词、角色声音区分、张力 | 给两个角色档案+冲突点 |
| 4 | 情感戏 | 展示而非告知、克制 | 给关系+情绪转折点 |
| 5 | 章尾钩子 | 钩子吸引力、不空泛预告 | 给本章已写内容，要求收尾 |
| 6 | 长上下文续写 | **逻辑一致性**、伏笔接续 | 给多章前文+角色状态+已知事实清单 |
| 7 | 指定文风模仿 | 风格可控性 | 给一段风格范本，要求模仿续写 |

### 单个场景的数据结构
```jsonc
{
  "id": "scene_06_consistency",
  "type": "long_context_continuation",
  "title": "长上下文一致性测试",
  "input": {
    "context_prompt": "（多章前文摘要、角色档案、世界观）",
    "user_prompt": "接着最后一段往下写约 1500 字",
    "target_chars": 1500
  },
  // 用于自动查"逻辑严谨"：生成结果若违反这些事实即扣分
  "known_facts": [
    { "fact": "李慕白的右臂在第30章已被斩断", "type": "state" },
    { "fact": "玄铁令已交给苏清欢保管", "type": "possession" },
    { "fact": "目前是寒冬，角色身处雪山", "type": "setting" }
  ],
  // 该场景重点关注的维度（不同场景侧重不同）
  "focus_dimensions": ["consistency", "prose", "creativity"]
}
```

### 存放与版本化
- 路径：`tools/writing-eval/scenarios/*.json`，每个场景一个文件。
- 进 git 版本控制。**场景一旦定稿就冻结**，改场景等于换尺子，历史不可比。
- 真实感来源：建议从你自己写得满意的章节里抽取改编，比凭空造的更贴合实战。

## 四、打分器设计（Scorers）

分两层。**A 层每次必跑（免费），B 层里程碑才跑（花钱）。**

### A 层 — 客观自动指标（纯代码，零 token 成本）

| 指标 | 算法 | 越高越好? | 数据来源 |
|---|---|---|---|
| **AI腔密度** | 命中"一级禁用词"次数 / 字数 × 1000 | 越低越好 | 复用现有提示词里的禁用词表 |
| **套话句式命中** | 正则匹配"不是A而是B""嘴角勾起一抹"等 | 越低越好 | 维护一份句式正则库 |
| **n-gram 重复率** | 重复的 2/3-gram 占比 | 越低越好 | 代码统计 |
| **用词丰富度(TTR)** | 不重复词数 / 总词数 | 越高越好 | 分词后统计 |
| **字数达标度** | 1 - \|实际-目标\|/目标 | 越接近1越好 | 对比 target_chars |
| **格式违规** | 是否出现分隔符/Markdown/章节号/元叙事 | 0违规最好 | 正则黑名单 |
| **弱副词密度** | "微微/淡淡/缓缓/轻轻"每千字次数 | ≤3/千字 | 词表统计 |
| **对话占比** | 引号内字数 / 总字数 | 参考值(无绝对好坏) | 统计 |
| **句长方差** | 句子长度标准差（节奏代理） | 适中（太低=呆板） | 统计 |
| **一致性硬伤** | 用 known_facts 逐条检查是否被违反* | 0违反最好 | 见下 |

> *一致性硬检：A 层先做"关键词级"粗检（如事实提到"右臂断了"，就检查输出里有没有"右臂"做出矛盾动作的描述）。精细判断留给 B 层 AI 裁判。A 层只抓最明显的硬伤。

**A 层产出**：每份输出一组数值 + 每个指标的"命中明细"（具体哪句话触发了套话扣分，方便你定位）。

### B 层 — AI 裁判指标（强模型，按需跑）

裁判模型选择铁律：
- **用比被测模型更强的模型当裁判**（如被测 DeepSeek，裁判用 Claude/Qwen-Max）。
- **裁判 ≠ 被测同一模型**，避免"自我偏好"虚高。

| 维度 | 怎么判 |
|---|---|
| 文笔质感 | 盲测 A/B：两版输出哪个文字更好 |
| 创意/反套路 | 盲测 A/B：哪个情节与措辞更新颖 |
| 角色声音一致性 | 给角色档案，判对话是否贴合人设、两角色是否可区分 |
| 章尾钩子吸引力 | 判结尾是否让人想翻下一章、是否空泛预告 |
| **逻辑一致性** | 给 known_facts，judge 逐条核对输出有无矛盾，输出违反清单 |
| 整体可读性 | 盲测 A/B 综合判断 |

**B 层两种模式**：
- **A/B 盲测模式**（推荐，主力）：给裁判两版输出（随机打乱顺序，去掉标识），问"哪个更好+理由"。产出胜率。
- **Rubric 打分模式**（辅助）：给单份输出按维度打 1-5 分 + 理由。绝对分不稳，仅作趋势参考。

### 防裁判偏见的措施
1. **顺序随机化**：A/B 两版位置随机，破除"位置偏好"。
2. **长度对齐**：控制两版字数接近，或明确指示裁判"不要因为长而给高分"。
3. **多次投票**：同一对 A/B 让裁判判 3 次取多数，降低裁判自身随机性。
4. **要求给理由**：强制裁判先说理由再下结论，提升判断质量。

## 五、评分汇总与成绩单（Report）

### 单项 → 综合分
1. 每个指标先**归一化到 0~100**（如 AI腔密度：0次/千字=100分，≥5次=0分，线性映射）。
2. 同一场景 N 次生成取**中位数**（抗异常值）。
3. 各场景按维度汇总，再按**权重**合成总分。权重可配，建议默认：

| 维度组 | 权重 | 说明 |
|---|---|---|
| 去AI腔（AI腔+套话+重复+弱副词） | 30% | 你当前最痛的点，给高权重 |
| 逻辑一致性 | 25% | 长篇命门 |
| 文笔质感（B层） | 20% | |
| 创意/反套路（B层） | 15% | |
| 指令遵循（字数+格式） | 10% | |

> 权重存配置文件，可随你关注点调整。总分只是"一眼看趋势"，**真正决策看分维度涨跌**。

### 成绩单数据结构
```jsonc
{
  "runId": "run_20260607_temp08_v3",
  "config": { "model": "deepseek-chat", "temperature": 0.8,
              "presencePenalty": 0.3, "promptVersion": "v3", "label": "温度0.8+提示词v3" },
  "createdAt": "2026-06-07T12:00:00Z",
  "totalScore": 72.4,
  "byDimension": {
    "ai_slop": 81, "consistency": 68, "prose": 70, "creativity": 65, "instruction": 88
  },
  "byScenario": [
    { "scenarioId": "scene_01_opening", "scores": { "...": 0 },
      "samples": [ { "text": "...", "metrics": {}, "hits": ["套话:嘴角勾起 @ 第3句"] } ] }
  ]
}
```
- 存 `tools/writing-eval/reports/run_*.json`，每次跑测一份，**永久留档**。
- 留档是关键：没有历史就无法对比"这次比上次强多少"。

### 对比（Comparison）
- 选任意两份 Report（通常是"改动前 vs 改动后"），逐维度算差值。
- 输出："去AI腔 +13、一致性 -2、创意 +8、总分 +6.4"，并高亮**回退项**（变差的维度要警惕）。

### 界面设计要点
- **颜色编码**：🟢进步 / 🔴回退 / ⚪持平，一眼扫到回退项。
- **可下钻**：总分→维度→场景→单份输出→具体扣分句子，逐层定位问题。
- **并排对比**是核心交互：永远是"新 vs 旧"，不是孤立看一份。
- **扣分可解释**：每个扣分都标出"哪句话触发的"，否则你不知道怎么改。

## 七、技术实现选型

### 放在哪
独立目录 `tools/writing-eval/`，与正式产品代码隔离，互不影响：
```
tools/writing-eval/
├─ scenarios/            # 固定测试场景(json), 版本化
├─ reports/              # 历次成绩单(json), 留档
├─ prompts/              # 被测的提示词版本快照(v1/v2/v3…)
├─ src/
│  ├─ runner.ts          # 跑测调度: 读场景→调AI生成→收集输出
│  ├─ scorers/
│  │  ├─ auto/           # A层自动指标(每个指标一个纯函数)
│  │  └─ judge/          # B层AI裁判(调强模型)
│  ├─ aggregate.ts       # 归一化+加权→成绩单
│  └─ server.ts          # 起本地 web 服务，喂数据给界面
└─ ui/                   # 可视化界面(单页应用)
```

### 用什么技术
- **复用你现有的技术栈**：你前端是 Vue + TS，界面就用 Vue 单页，零学习成本。
- **生成调用**：直接复用你已有的 `/ai/prompt` 接口或 DeepSeek 客户端逻辑，**用作者自己的 key**，不走用户计费。
- **A 层指标**：纯 TypeScript 函数，输入文本输出数值，可单元测试。中文分词可用轻量库（如 `nodejira`/`segment`）或简单按标点+字符切。
- **B 层裁判**：独立模块，调一个强模型 API，结构化输出（要求返回 JSON）。
- **存储**：全用本地 json 文件，不需要数据库。简单、可 diff、可进 git。
- **界面起法**：`server.ts` 起个本地端口读 reports/ 目录，浏览器打开即用。

### 与现有资产的复用点
| 现有资产 | 复用到 |

## 八、分期落地计划

**严格按阶段来，每阶段都能独立产生价值，不必等全做完。**

### 阶段一：自动指标 MVP（性价比最高，先做这个）
- 准备 3~4 个场景（开篇、对话、章尾钩子、长上下文一致性）。
- 实现 A 层核心打分器：AI腔密度、套话句式、重复率、字数达标、格式违规。
- 跑测脚本：一条命令跑完，输出一份 json 成绩单 + 命令行表格。
- **此时还没界面**，但已经能回答"AI腔降了没"。
- 产出：改提示词后 1 分钟内得到客观数据。**零 token 成本。**

### 阶段二：可视化界面
- 起本地 web 服务 + Vue 单页。
- 实现页面 1（仪表盘趋势）、页面 4（并排对比）。
- 成绩单留档 + 历史对比。
- 产出：可视化看趋势、并排对比版本。

### 阶段三：AI 裁判（B 层）
- 接入强模型裁判，实现 A/B 盲测模式（顺序随机、多次投票、要理由）。
- 补全页面 2 的"裁判开关 + 对照组选择"、页面 4 的胜率与裁判理由展示。
- 产出：文笔/创意这类软维度也能量化对比。

### 阶段四：打磨（按需）
- 页面 3 实时进度、维度权重可调、场景管理界面、消耗预估与上限保护。

## 九、使用流程（做完之后你的日常）

1. 改了提示词/参数 → 在界面"新建跑测"，填标签、选对照组。
2. 点开始，等几十秒到几分钟（看是否开 B 层裁判）。
3. 看成绩单：总分涨了？哪个维度涨、哪个退？
4. 下钻到退步的场景，看具体哪句话扣分。
5. 据此再改 → 再测。**形成"改—测—对比"的闭环。**

## 十、风险与注意事项

1. **裁判不是真理**：AI 裁判有偏见，只看"相对趋势"，别迷信绝对分。关键改动最终仍需你人眼抽查。
2. **场景会过时**：固定场景用久了可能"过拟合"（你专门照着指标改提示词，分高了但实际没变好）。每隔一段时间补充新场景，但**旧场景保留**以维持可比性。
3. **成本控制**：B 层裁判烧 token，设单次跑测消耗上限，默认关闭、按需开。
4. **只测不等于变强**：测试系统告诉你"变了没"，但"怎么变强"还得靠你的提示词/流程设计。它是尺子，不是工匠。
5. **别接入用户端**：这是开发工具，消耗作者自己的 key，永远不要让它出现在用户客户端里。
