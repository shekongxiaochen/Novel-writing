# AI 漫剧（小说 → 逐帧动画短剧）设计方案

> 状态：设计阶段 · 分支 `feature/ai-comic`
> 本文档记录功能目标、技术路线、AI 剧组（多智能体）架构与现实约束，作为开发蓝图。

---

## 0. 一句话目标

在现有小说软件的基础上，让用户为**角色 / 物品 / 场景**建立"描述 + 形象"的素材库；AI 读小说自动编排剧本（给角色、物品分配戏份，定场景），再调用这些素材分段生成视频，产出**逐帧动画短剧**。用户以"总导演"身份只在关键节点审片拍板。

### 0.1 这是「我自己的软件」，不是「即梦套壳」

核心立场：**做给用户用的产品，用户不需要懂、也不需要直接碰即梦/可灵。** 软件在内部调用生图/生视频能力（开放接入，可换任意厂商），但对用户呈现的是"一键把我的小说变成动画短剧"。

**护城河 = 用户在本软件里积累的整套结构化世界**（角色、物品、场景、关系、大纲、伏笔、正文）。即梦/豆包这类通用工具做不到自动编排，因为它们没有这份数据；只有本软件知道"张三是谁、那把剑长什么样、书院在哪、剧情怎么走"，所以能"按图索骥"自动成剧。

> 红线自检：若某功能最终只是"把用户的文字转发给即梦"，那用户自己去用即梦即可，等于白做。每个功能都必须利用"结构化世界数据"做出用户单用工具做不到的事（自动记忆、自动编排、自动保持连贯、自动质检）。

### 0.2 三层架构

```
【素材层】用户在软件里建好（角色/物品本就有描述，补"形象"；场景为新增）
   角色 = 描述 + 形象（多视图立绘，每视图带描述）
   物品 = 描述 + 形象（多视图，每视图带描述）
   场景 = 描述 + 形象（新增功能）
        ↓
【编排层】AI 当导演（产品核心价值）
   读小说 → 剧本/分镜
   ├─ 这场戏用哪些角色（调其形象）
   ├─ 用到哪些物品（调其形象）
   ├─ 在哪个场景（调其形象）
   └─ 剧情 / 台词 / 动作 / 镜头 / 每镜头用哪个视图
        ↓
【生成层】素材齐备 → 分段「图 + 文 → 视频」→ 质检接力 → 拼接成片
```

## 1. 现实定位与天花板（必读，别忘）

这一节是给未来的自己和协作者的"清醒剂"。开发中任何时候觉得"效果不够番剧级"，回来看这里。

### 1.1 已确认的技术组合

| 维度 | 选择 | 含义 |
|------|------|------|
| 成品形态 | 逐帧动画短剧（角色会动） | 不是静态漫画 |
| 素材建模 | 角色/物品/场景均「描述 + **多视图形象**」 | 2D 设定稿，非 3D 模型 |
| 形象来源 | **直接接 AI 文生图生成**（第一版不做手动上传） | 开放接入，用户自填 Key |
| 动作来源 | **AI 生成** | 不准备动捕 / 真人参考视频 |
| 生成接入 | 开放接入任意厂商（推荐名单：即梦 Seedream / Nano Banana Pro 等经验证能锁角色的） | 不绑死任何一家、不暴露厂商给用户 |
| 用户角色 | 总导演，只审关键点 | 中间环节自动跑 |

### 1.2 这个组合的真实天花板

- **多视图立绘 ≠ 3D 模型**：立绘是"一组定死的参考图"，AI 要画立绘里没有的姿势时仍需"想象+重画"，角色一致性**只能大体锁住、锁不死**。
- **AI 生成动作是最大短板**：复杂动作（打斗、杂技、大幅肢体）易翻车（鬼畜、穿模、多指）。简单动作（说话、转身、走近、表情、环境镜头）才稳。
- **逐帧视频是接力赛**：一次只能生成几秒，要一段段拼。三大连贯性（角色 / 镜头 / 风格）全靠 agent 维护。
- **配音口型对不齐**：角色大特写说话时口型难对，是现阶段硬伤 → 用旁白/内心独白/中远景规避。

**正确预期**：能做出"有模有样的 AI 动漫短剧"，达不到"番剧级零漂移流畅动画"。skill/多智能体能拉高**质量下限和稳定性**，但**拉不高底层模型的能力上限**——模型画不好的，再多返工也变不出来，agent 的作用是"在能力范围内逼出最稳结果，并自动绕开做不到的"。

### 1.3 成本现实（开发计费模型时参考）

| 形态 | 用什么模型 | 一集（约3分钟）含试错的粗估 |
|------|-----------|--------------------------|
| 静态画面 / 关键帧 | 即梦图片（约 $0.03/张） | 几十 ~ 一百多元 |
| **逐帧动画**（本方案目标） | 视频模型（约 $0.1~0.5/秒） | **几百 ~ 两千元+** |

> 视频生成是烧钱大头。计费务必沿用现有"用户自填 API Key"模式，或按集预扣费，避免平台兜底。火山引擎官方即梦企业 API 实际价格需到控制台核实（可能有阶梯/额度）。

## 2. 核心技术路线：图 + 文 → 视频

用户确认的完整链路：**文生图固定角色 + 文字当剧本 + 「图(角色多视图) + 文(剧本镜头)」喂视频模型生成视频。**

```
① 文生图 → 角色多视图立绘，每个视图带描述（谁 / 什么角度 / 外貌特征 / 服饰）
   —— 这就是"建模/定妆"，把角色长相固定下来
② 小说剧情 → 剧本 / 分镜（演什么 / 景别 / 动作 / 台词 / 该用哪个视图）
   —— 文字当剧本
③ 每镜头先生成"关键帧图"（文生图 / 图生图，带对应视图参考）
   —— 画面在便宜的图片阶段定死、可反复调
④ 「关键帧图 + 对应角色视图 + 剧本镜头描述」→ 图文生视频
   —— 让定好的画面按剧本动起来（贵，放最后）
⑤ agent 自动质检 + 自动接力 → 不合格自动重生成；首尾帧串连防跳戏
⑥ 配音 + 字幕 + 音效 + 拼接 → 成片
```

**硬性技术前提**：第 ④ 步的视频模型**必须支持"把角色图当参考喂进去"（图生视频 / 图文生视频）**，而不是只能"打字出视频"的纯文生视频。否则喂进去的角色图它不认，出来仍会变脸。即梦、可灵等支持此能力。

**设计精髓**：把"控制"放在便宜的图片阶段（①③，几毛一张可反复调），把"花钱"压到最后视频阶段（④）。画面定死后才让它动，省钱又可控。

### 2.1 三类素材都是「描述 + 多视图形象」（已确认方案）

角色、物品、场景统一建模：不是一张图，而是**一套多视图形象，每个视图自带描述**。

```
角色「张三」
  ├─ 正面：图 + 描述「张三正面，青年男性，黑色短发，粗布长衫，眉目坚毅」
  ├─ 侧面：图 + 描述「张三侧面，露出左脸刀疤」
  ├─ 背面：图 + 描述「张三背面，发髻，长衫后摆」
  └─ 表情：图 + 描述「张三·愤怒 / 张三·微笑」

物品「断水剑」
  ├─ 主视图：图 + 描述「青锋长剑，剑身有水波纹，红缨」
  └─ 细节/握持：图 + 描述「张三握持断水剑的样子」

场景「青云书院」
  ├─ 全景：图 + 描述「白墙黛瓦的书院，清晨薄雾」
  ├─ 门口：图 + 描述「书院正门，挂‘青云’匾额」
  └─ 内院：图 + 描述「庭院，石桌石凳，老槐树」
```

- 每个视图描述要**标明这是谁/什么物/什么地、什么角度、什么特征**，并能与剧本镜头对应
- 三类素材都源自现有/新增档案：角色、物品**已有文字描述**（补"形象"即可）；**场景为新增功能**
- 视频生成时，分镜师为每个镜头**指定该用哪些素材的哪个视图**（正面镜头用正面视图、握剑镜头调物品握持图、书院戏调对应场景图）
- 喂多视图（而非单图）让模型对素材理解更全，换角度/换镜头时更不易跑样

### 2.2 形象怎么来：直接接 AI 文生图（第一版）

- 第一版**不做手动上传**，直接由 AI 文生图生成形象：把档案的文字描述组织成画图提示词 → 调生图 API → 出多视图。
- 角色/物品/场景的"描述"字段就是生图提示词的来源，**描述写得越准，形象越贴**。
- 用户在关键节点挑版本/微调描述重生成，满意后**锁定**存档。

### 2.3 接入方式：开放接入，不绑死、不暴露厂商

- **沿用现有"用户自填 API Key / Base URL / Model"模式**（与小说 AI 一致），漫剧的生图与生视频同样开放接入，**不绑死即梦**。
- 即梦 Seedream / Nano Banana Pro 等只是**经验证能"参考图锁角色"的推荐名单**，非唯一选项。
- 界面需**标注能力门槛**：所选模型必须支持"参考图锁角色 / 图生视频"，否则提示"角色可能不一致"。

## 3. AI 剧组（多智能体）架构

整个功能的"大脑"。把一个动画工作室"装进 AI"，每个角色是一段 skill（职责 + 标准 + 产出格式）。

### 3.1 角色分工

| 角色 | 职责 | 输入 → 产出 | 把关对象 | 成本 |
|------|------|------------|---------|------|
| **编剧** | 理清这一集演什么 | 正文+大纲 → 剧情大纲、情绪曲线 | — | 低（纯文本）|
| **分镜师** | 拆镜头并**编排素材** | 剧情 + **素材库（角色/物品/场景）** → 分镜表（每镜头：用哪些角色/物品、在哪个场景、各自用哪个视图、景别/动作/台词）| 编剧产出 | 低 |
| **关键帧画师** | 生成每镜头第一帧静态画 | 角色/物品/场景视图 + 景别 → 关键帧图 | — | 中（图片）|
| **动画师** | 关键帧图生视频 | 关键帧 + 视图参考 + 动作描述 → 动态片段 | — | **高（视频）**|
| **作画监督**（质检官）| 挑毛病、打回重做 | 成片 → 合格/打回+理由 | 画师、动画师 | 中（视觉评估）|
| **导演**（总协调）| 全局连贯/节奏/风格、返工决策 | 全流程 → 调度决策 | 所有人 | 中 |

> **编排层的核心在"分镜师"**：它不是凭空写镜头，而是**从素材库里点名**——这场戏调哪些角色、哪些物品、哪个场景，每个用哪个视图。这一步把"用户积累的世界"和"要生成的画面"绑定起来，是产品护城河的落点（见 0.1）。

### 3.2 关键角色：作画监督 + 导演

这两个**把关角色**是"强 agent"的核心，是把"人工抽卡筛选"自动化的执行者。没它们 = AI 自嗨；有它们 = 强 agent。

- **作画监督**：用 AI 视觉能力对比"这帧的角色 vs 锁定立绘"，打分；检查动作有无鬼畜/穿模/多指；低于阈值自动打回重生成（限最多 N 次，防死循环）。
- **导演**：把控整集风格统一、节奏、镜头衔接；决定哪些镜头返工、哪些降级处理；在"模型反复做不到"时改策略（如把打斗改成快切静态冲击帧）。

### 3.3 扬长避短（写进分镜师 / 导演的 skill）

既然 AI 动作短板在"复杂动作"，则：
- 分镜阶段**主动多生成 AI 好实现的镜头**：说话、转身、走近、表情特写、环境空镜。
- **少设计高难动作**；复杂动作用"快切 + 多个静态冲击帧"表现（动画业经典手法"有限动画"）。
- 说话镜头**优先旁白/内心独白/中远景**，规避口型难题。

## 4. 用户交互：总导演只审关键点

自动流水线，仅在关键节点要"总导演"拍板：

1. **素材定妆确认**：AI 生成角色/物品/场景的多视图形象 → 用户挑版本/改描述重生成 → 锁定存档。（一致性地基，必须人工拍板）
2. **成片审查**：整集自动产出后 → 用户审 → 个别不满意的镜头手动让 agent 重做。

中间环节（剧本、分镜编排、关键帧、视频、质检）全自动；但应**可展开查看/干预**（高级用户想插手时能进去）。

## 5. 数据模型（草案，开发时细化）

复用现有"作品 → 角色/大纲"结构，新增漫剧相关实体。建议存于 localStorage（与现有一致）+ 可选云同步。

```
ComicProject（漫剧项目，挂在某个 novel 下）
  - id, novelId, title, status, createdAt

// ── 素材层：角色/物品/场景统一用「描述 + 多视图」结构 ──

AssetView（单个视图，三类素材共用）
  - id
  - kind: 'front' | 'side' | 'back' | 'detail' | 'expression' | 'establishing' | ...
  - imageUrl: string        // 形象图
  - description: string     // 这是谁/什么物/什么地、角度、特征，供分镜与视频对应
  - prompt: string          // 生成此视图所用的提示词（便于重生成/微调）
  - emotion?: string        // 角色表情视图用

CharacterModel（扩展现有角色档案）
  - characterId            // 关联现有角色（复用其文字描述）
  - views: AssetView[]
  - locked: boolean
  - stylePrompt: string    // 画风/外貌补充提示词

ItemModel（扩展现有物品档案）
  - itemId                 // 关联现有物品（复用其文字描述）
  - views: AssetView[]
  - locked: boolean
  - stylePrompt: string

SceneModel（新增：场景档案，本身也是一类设定条目）
  - id, novelId
  - name, description      // 场景文字描述（新增的档案字段）
  - views: AssetView[]
  - locked: boolean
  - stylePrompt: string

// ── 编排层 ──

Storyboard（分镜表，对应一集/一章）
  - id, comicProjectId, chapterId
  - shots: Shot[]

Shot（单个镜头）
  - id, order
  - sceneId                          // 本镜头在哪个场景
  - sceneViewId                      // 用该场景的哪个视图
  - characterIds: [...]              // 出场角色
  - itemIds: [...]                   // 出场物品
  - assetViewIds: [...]              // 每个出场角色/物品用哪个视图
  - shotType: 'closeup' | 'medium' | 'wide' | 'establishing'
  - action: string                  // 动作描述
  - dialogue: string                // 台词/旁白
  - emotion: string
  - keyframeUrl: string             // ③ 生成的关键帧图
  - clipUrl: string                 // ④ 生成的视频片段
  - qcStatus: 'pending' | 'pass' | 'retry' | 'failed'
  - qcReason: string                // 作画监督的评语
  - retryCount: number

// ── 生成层 ──

ComicRenderJob（渲染任务，追踪 agent 流水线进度）
  - id, comicProjectId, stage, progress, costEstimate, log[]
```

## 6. 后端 / API 集成（草案）

- **生图/生视频调用**：开放接入，新增图像与视频两类 provider 配置；**厂商对用户不暴露**，用户看到的是"生成形象/生成视频"，内部可切换即梦/可灵等。Key 来源（平台统一 vs 用户自填）见第 9 节待定。
- **异步任务**：视频生成慢（几分钟/段），需任务队列 + 进度轮询 + 断点续跑（复用现有 Rust 后端思路，或前端任务管理）。
- **成本预估与预扣**：开工前给用户算"这一集大概多少钱/多少额度"，确认后再跑。
- **失败重试**：作画监督触发的重生成要有上限和降级策略。

### 6.1 Provider 配置必须后台可管，严禁写死（硬约束）

生图 / 生视频的 `base_url / api_key / model / 价格 / 倍率` 等**一律不得硬编码**，必须做进**后台管理界面**，运营可在线增删改、切换激活，与现有 AI 模型配置一致。

**复用现有模式**（已验证）：
- 参照 `backend/src/admin/ai_config.rs`（AI 模型后台）与 `embedding_config.rs`（嵌入模型后台）
- 服务层参照 `backend/src/services/ai_provider_service.rs`（DB 存储 provider + 激活项）
- 新增两类 provider 表与后台页：
  - **图像 provider**（`image_providers`）：文生图 / 图生图 / 多视图生成
  - **视频 provider**（`video_providers`）：图文生视频
- 每类支持：多条配置、标记激活项、价格/倍率字段（接入现有钱包计费）、在线测试连通性
- 前端只调用后端统一接口（如 `/comic/gen-image`、`/comic/gen-video`），**前端不出现任何厂商 URL / Key**

> 原则：换厂商、改 Key、调价格，全部后台点几下完成，不需要改代码、不需要重新打包。

## 7. 分阶段实施建议

每阶段都是可独立验证、可独立交付的里程碑。顺序按"先打地基（素材层）、再验证生成、最后做全流水线"安排。

| 阶段 | 内容 | 验证目标 | 花钱 |
|------|------|---------|------|
| **P0 生成能力实测** | 接一个生图/生视频 API，手动跑「文字→多视图→图文生视频」 | 眼见为实：角色锁不锁得住、图生视频效果、成本 | 极少 |
| **P1 场景档案** | 新增"场景"功能（描述+增删改查，跟角色/物品平级） | 补齐素材层缺的一类 | 0 |
| **P2 素材形象** | 给角色/物品/场景接 AI 文生图，生成多视图（每视图带描述）+ 定妆锁定 UI | 素材层完整：三类素材都有"描述+形象" | 少（图片）|
| **P3 剧组大脑**（纯文本）| 编剧 + 分镜师，读小说 → 分镜表，并从素材库点名角色/物品/场景 | 验证"AI 自动编排"这一核心价值，几乎不花钱 | 极少 |
| **P4 关键帧** | 每镜头出关键帧图 + 作画监督质检 | 静态画面质量与一致性 | 中 |
| **P5 逐帧视频** | 图文生视频 + 接力 + 配音字幕拼接 | 完整成片 | 高 |

**为什么这个顺序**：素材层（P1/P2）是后面一切的地基——没有角色/物品/场景的形象，AI 无法编排、更无法生成；且素材层做完即使不生成视频，"给角色/物品/场景配图的可视化设定集"本身对写小说就有价值。P0 单独先行，是在投入素材层开发前先确认底层生成能力够不够。

## 8. 风险登记

| 风险 | 影响 | 缓解 |
|------|------|------|
| AI 动作翻车（复杂动作） | 成片质量 | 分镜扬长避短；作画监督质检返工；有限动画手法 |
| 角色变脸（立绘锁不死） | 一致性 | 多视图立绘按角度选基准；关键帧阶段反复调；视觉打分 |
| 镜头跳戏 | 连贯性 | 首尾帧自动接力 |
| 口型对不齐 | 出戏 | 多旁白/中远景，少大特写说话 |
| 视频生成烧钱 | 成本失控 | 用户自填 Key；开工前成本预估预扣；重试上限 |
| 底层模型上限 | 达不到番剧级 | 正确预期；agent 只在能力内逼出最稳结果 |
| 视频生成慢 | 体验 | 异步任务+进度+断点续跑 |
