# 认证后端设计方案

## 1. 目标

本项目最终形态是桌面写作软件，后端主要承担这几类职责：

1. 用户注册、登录、退出登录
2. 用户基础资料存储
3. 后续云同步、订阅、跨设备数据归属
4. 必要时承接邮件验证码、找回密码、安全审计

当前阶段不追求“大而全”，重点是先做出一套：

- 本地开发能跑
- 测试环境可用
- 后续切正式服务器时不用推翻重来

## 2. 当前代码现状

### 后端

仓库仅保留 **Rust 后端**（目录 `backend/`，默认端口 `8080`）：

- `backend/src/handlers/auth.rs`
- `backend/src/services/auth_service.rs`
- `backend/src/models/user.rs`、`auth_session.rs`

已具备（与原先 FastAPI 设计对齐的接口）：

- 邮箱验证码发送 `POST /auth/send-code`
- 邮箱注册 `POST /auth/register`
- 邮箱密码登录 `POST /auth/login`
- 当前用户 `GET /auth/me`
- 退出登录 `POST /auth/logout`

数据库核心表：

- `users`
- `email_verification_codes`
- `auth_sessions`

### 前端

前端当前仍是本地认证：

- [frontend/src/lib/auth.ts](D:\Novel writing\frontend\src\lib\auth.ts)
- [frontend/src/composables/useAuth.ts](D:\Novel writing\frontend\src\composables\useAuth.ts)

它把用户和会话直接存在 `localStorage`，还没有接后端接口。

这意味着现在存在两套认证体系：

1. 前端本地假认证
2. 后端真实认证

下一阶段要做的第一件事，不是继续堆接口，而是让前端改为只信任后端。

## 3. 设计原则

### 3.1 先做邮箱密码，不做短信

现阶段最适合的账号体系是：

- 注册：邮箱 + 验证码 + 密码
- 登录：邮箱 + 密码

原因：

- 成本低
- 接入简单
- 合规负担比短信轻
- 对桌面软件足够用

### 3.2 会话优先，不急着上复杂 OAuth

当前项目不需要一开始就做：

- 微信登录
- QQ 登录
- Apple 登录
- 多租户 SSO

先把最基本的第一方账号体系打稳。

### 3.3 明确区分“认证”与“同步”

认证系统只负责：

- 你是谁
- 你是否已登录
- 当前 token 是否还有效

不要把“小说数据同步策略”混在第一版认证里。否则容易把一个小问题做成一整团。

## 4. 推荐的第一阶段架构

### 4.1 技术选型

沿用现有后端栈：

- API：FastAPI
- ORM：SQLAlchemy
- DB：MySQL（当前代码已按 MySQL 配置）
- 邮件：SMTP

不建议现在切技术栈。现有代码已经起步，继续往前推更省力。

### 4.2 认证方式

第一阶段建议采用：

- `access token` 风格的随机会话令牌
- 服务端保存会话记录
- 客户端通过 `Authorization: Bearer <token>` 调用接口

也就是“数据库会话令牌”方案，而不是一上来就全 JWT 化。

原因：

- 当前后端已经是这个方向
- 服务端可强制失效 token，方便退出登录和封禁
- 实现比 refresh token + JWT 双令牌更直接

### 4.3 客户端保存位置

桌面端建议保存：

- `token`
- `expires_at`
- `user`

正式版最好放系统安全存储：

- Windows Credential Manager
- macOS Keychain

开发阶段可以先临时放本地配置，但不要继续沿用浏览器 `localStorage` 那套假认证。

## 5. 第一阶段功能范围

### 必做

1. 邮箱验证码注册
2. 邮箱密码登录
3. 获取当前用户
4. 退出登录
5. 会话过期校验

### 建议尽快补上

1. 忘记密码
2. 重置密码
3. 验证码发送限流
4. 登录失败简单限流
5. 会话清理任务

### 暂缓

1. 短信登录
2. 第三方登录
3. 双因素认证
4. 设备管理页

## 6. 数据模型设计

## 6.1 users

建议保留并逐步补齐：

- `id`
- `email`
- `password_hash`
- `display_name`
- `is_active`
- `email_verified`
- `created_at`
- `updated_at`

建议新增：

- `last_login_at`
- `last_login_ip`
- `password_changed_at`

用途：

- 审计最近登录
- 密码修改后可让旧会话失效

## 6.2 auth_sessions

当前已有：

- `id`
- `user_id`
- `token`
- `expires_at`
- `created_at`
- `last_seen_at`

建议新增：

- `revoked_at`
- `client_type`：`desktop` / `web` / `unknown`
- `device_name`
- `ip`
- `user_agent`

说明：

- 第一阶段即便前端不上传完整设备信息，也建议把字段预留好
- 将来做“查看登录设备”“踢下线”会轻松很多

## 6.3 email_verification_codes

当前模型可继续用，但建议补两类信息：

- `sent_at`
- `request_ip`

建议限制：

- 同邮箱 60 秒内最多发 1 次
- 同邮箱 10 分钟内最多保留最近若干条有效验证码

## 7. 接口设计

建议第一阶段保留并规范为以下接口。

### 7.1 发送注册验证码

`POST /auth/send-code`

请求：

```json
{
  "email": "user@example.com",
  "purpose": "register"
}
```

返回：

```json
{
  "message": "验证码已发送"
}
```

规则：

- 邮箱格式校验
- `purpose=register` 时，如邮箱已注册则报错
- 增加发送频率限制

### 7.2 注册

`POST /auth/register`

请求：

```json
{
  "email": "user@example.com",
  "password": "12345678",
  "code": "123456",
  "display_name": "阿青"
}
```

返回：

```json
{
  "token": "xxx",
  "expires_at": "2026-05-20T10:00:00Z",
  "user": {
    "id": "xxx",
    "email": "user@example.com",
    "display_name": "阿青",
    "created_at": "2026-05-20T09:50:00Z"
  }
}
```

建议继续保留“注册成功即登录”。

### 7.3 登录

`POST /auth/login`

请求：

```json
{
  "email": "user@example.com",
  "password": "12345678"
}
```

返回结构同注册。

建议补充：

- 登录成功时更新 `last_login_at`
- 记录本次会话的设备信息和 IP

### 7.4 当前用户

`GET /auth/me`

请求头：

```http
Authorization: Bearer <token>
```

返回：

```json
{
  "id": "xxx",
  "email": "user@example.com",
  "display_name": "阿青",
  "created_at": "2026-05-20T09:50:00Z"
}
```

### 7.5 退出登录

`POST /auth/logout`

行为：

- 删除当前会话，或标记 `revoked_at`

### 7.6 忘记密码

建议新增：

- `POST /auth/send-code`，`purpose=reset_password`
- `POST /auth/reset-password`

`reset-password` 请求建议为：

```json
{
  "email": "user@example.com",
  "code": "123456",
  "new_password": "new-password"
}
```

执行规则：

- 校验验证码
- 更新密码
- 删除或失效该用户所有旧会话

## 8. 安全设计

## 8.1 密码哈希

当前实现使用 `pbkdf2_sha256`，可继续跑通开发版，但正式版建议升级到：

- `argon2id` 优先
- 或 `bcrypt`

原因：

- 现成库更成熟
- 参数调优更清晰
- 更适合长期维护

建议做法：

1. 新密码统一使用 `argon2id`
2. 保留对旧 `pbkdf2_sha256` 的校验兼容
3. 用户下次成功登录时自动升级哈希

## 8.2 验证码

建议规则：

- 6 位数字
- 10 分钟有效
- 单次使用
- 校验成功立即 `consumed_at`

并增加：

- 同邮箱发送冷却
- 同 IP 发送频率限制
- 连续输错次数限制

## 8.3 会话有效期

当前配置是 `30` 天。

建议调整为：

- 开发阶段：7 到 30 天都可以
- 正式阶段：14 到 30 天

同时补一个“滑动续期”策略：

- 用户活跃调用接口时，若会话即将过期，可延长 `expires_at`

不一定第一版就做，但可以预留。

## 8.4 最小审计

建议后端至少记录：

- 用户注册时间
- 最近登录时间
- 最近一次会话更新时间
- 验证码发送时间

不需要一开始做完整风控，但这些最小记录值得保留。

## 9. 与桌面客户端的衔接

### 9.1 当前问题

当前前端的 `useAuth` / `lib/auth.ts` 仍是本地模拟认证。

这会带来两个问题：

1. 用户以为自己“注册成功”，但后端根本没有这个账号
2. 后续接云同步时，用户身份会断层

### 9.2 建议改法

前端认证分两层：

1. `authApi`
   - 专门请求后端 `/auth/*`
2. `authStore` / `useAuth`
   - 只管理当前登录状态

建议替换方向：

- 删除本地用户表逻辑
- 保留本地“当前 token / 当前 user”缓存
- 启动时调用 `/auth/me` 校验会话

### 9.3 桌面端存储建议

开发阶段：

- 可先保存在本地配置文件

正式阶段：

- 改为系统凭据存储

## 10. 实施顺序

建议按下面顺序推进。

### 第一步：统一认证来源

目标：

- 前端不再用 `localStorage` 假用户注册/登录
- 前端改走后端 `/auth/*`

### 第二步：补齐缺失能力

目标：

- 忘记密码
- 验证码限流
- 会话过期与清理

### 第三步：提升正式可用性

目标：

- 密码哈希升级到 `argon2id`
- 会话记录补设备信息
- 令牌存储迁移到更安全的位置

### 第四步：再考虑用户体系扩展

目标：

- 多设备管理
- 订阅绑定
- 云同步权限模型

## 11. 现阶段不建议做的事

1. 不建议先买正式中国内地服务器，只为做登录接口
2. 不建议现在上短信
3. 不建议一开始就把 JWT、refresh token、黑名单、设备信任全部堆齐
4. 不建议继续维持前端本地假认证和后端真认证双轨并存

## 12. 最终建议

对这个项目，最合适的第一版认证系统是：

- FastAPI + MySQL
- 邮箱验证码注册
- 邮箱密码登录
- 数据库会话令牌
- 前端只信后端认证结果

这套方案足够支撑：

- 内测
- 小范围真实用户使用
- 后续接入云同步

而且不会把你拖进过早的复杂基础设施里。

下一步建议直接开始做两件事：

1. 前端认证改接后端
2. 后端补 `reset-password` 与限流
