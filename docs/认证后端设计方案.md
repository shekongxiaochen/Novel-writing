# 认证后端设计方案

> 本文档描述 Lumen 流明**当前实际实现**的认证机制。后端为 Rust + Axum（目录 `backend/`，默认端口 `8080`），不再使用任何 Python / FastAPI 代码。

## 1. 目标

后端在桌面写作软件中承担以下职责：

1. 设备维度的账号自动注册与登录
2. 会话令牌签发与校验
3. 卡密充值与 AI 用量计费（钱包余额）
4. 云同步数据的用户归属

设计取向：**让用户零门槛使用**——无需填写邮箱、验证码或自设密码，打开即用；付费能力通过卡密充值挂接。

## 2. 认证模型：设备注册 + 自动账号

与传统"邮箱+密码+验证码"不同，本项目采用 **设备绑定 + 后端自动生成账号** 的模式。

### 2.1 核心流程

1. 客户端首次启动，本地生成并持久化一个设备标识 `X-Device-Id`
2. 客户端调用 `GET /auth/device-status` 查询该设备是否已有账号
3. 若无账号，调用 `POST /auth/register-by-device`：
   - 后端按设备哈希去重，自动生成随机 `username`（8 位）与 `password`（12 位）
   - 创建用户、初始化钱包、赠送 1 元体验额度
   - 返回 `username` / `password` / `token`，并立即登录
4. 客户端保存返回的账号密码与令牌；后续可用 `POST /auth/login` 以 `username` + `password` 重新登录

### 2.2 关键特征

- 用户**不感知**注册过程，账号密码由后端生成并由客户端保管
- `email` 字段存在但默认为空（`NULL`），`email_verified` 默认为 `0`——预留但当前不使用邮箱体系
- 一台设备对应一个账号，重复注册返回"本设备已注册"冲突

## 3. 接口清单（实际路由）

定义于 [backend/src/main.rs](../backend/src/main.rs)，处理函数在 [backend/src/handlers/auth.rs](../backend/src/handlers/auth.rs)。

### 公开接口（无需令牌）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/auth/device-status` | 查询设备是否已注册，需带 `X-Device-Id` 头 |
| POST | `/auth/register-by-device` | 按设备注册账号，需带 `X-Device-Id` 头 |
| POST | `/auth/login` | 用户名 + 密码登录 |

### 鉴权接口（需 `Authorization: Bearer <token>`）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/auth/me` | 获取当前用户信息（含余额） |
| POST | `/auth/logout` | 退出登录，删除当前会话 |

### 3.1 GET /auth/device-status

请求头：`X-Device-Id: <设备标识>`

返回：

```json
{
  "has_account": true,
  "username": "ab12cd34"
}
```

### 3.2 POST /auth/register-by-device

请求头：`X-Device-Id: <设备标识>`（无请求体）

返回 `201 Created`：

```json
{
  "username": "ab12cd34",
  "password": "Xy7kPq2mNw9R",
  "token": "<JWT>",
  "user": {
    "id": "...",
    "username": "ab12cd34",
    "display_name": "",
    "is_active": true,
    "balance_yuan": 1.0,
    "created_at": "2026-06-14T09:50:00Z"
  }
}
```

> 客户端必须妥善保存 `username` / `password`，这是该设备账号唯一的登录凭据。

### 3.3 POST /auth/login

请求：

```json
{
  "username": "ab12cd34",
  "password": "Xy7kPq2mNw9R"
}
```

返回：

```json
{
  "token": "<JWT>",
  "user": { "...": "同上 UserInfo" }
}
```

### 3.4 GET /auth/me

请求头：`Authorization: Bearer <token>`

返回 `UserInfo`（结构同上 `user` 字段）。

### 3.5 POST /auth/logout

请求头：`Authorization: Bearer <token>`

行为：从 `user_sessions` 删除当前令牌，并清除 Redis 会话缓存。

## 4. 数据模型（实际）

### 4.1 users

定义于 [backend/src/models/user.rs](../backend/src/models/user.rs)：

- `id`
- `username`
- `email`（`Option`，当前为空）
- `password_hash`
- `display_name`
- `device_id_hash`（设备哈希，用于去重）
- `registration_ip`
- `is_active`
- `email_verified`
- `created_at` / `updated_at`

### 4.2 user_sessions

会话表（注意：表名为 `user_sessions`）：

- `id`
- `user_id`
- `token`（JWT）
- `expires_at`
- `created_at`
- `last_seen_at`

### 4.3 钱包与计费

- `ai_wallets`：用户余额
- `ai_wallet_ledger`：流水
- `card_keys`：卡密
- 余额以 `balance_yuan`（元）对客户端呈现

## 5. 安全设计（实际实现）

### 5.1 令牌

- 使用 **JWT**（`jsonwebtoken`），由 `JWT_SECRET` 签发
- 同时在 `user_sessions` 表落库，并写入 Redis 缓存（`session:<token>`，TTL 60 秒）
- 校验时优先读缓存，未命中查库；缓存仅 60 秒，确保封禁/停用最多 60 秒后必然失效
- 默认有效期 `JWT_EXPIRATION_HOURS=720`（30 天）
- 退出登录从库中删除会话，服务端可强制失效

### 5.2 密码哈希

- 当前实现为 **PBKDF2**（`pbkdf2`，带随机 salt），迭代次数由 `PASSWORD_HASH_ITERATIONS` 控制，默认 `120000`
- 见 [backend/src/utils/crypto.rs](../backend/src/utils/crypto.rs)

### 5.3 限流与防刷

- **注册限流**：同一 IP 每小时最多注册 10 个账号（防止伪造大量 `X-Device-Id` 刷号）
- **登录失败限流**：按用户名与 IP 双维度计数，超阈值（`LOGIN_FAILURE_LIMIT`，IP 维度为其 5 倍）在 `LOGIN_FAILURE_WINDOW_MINUTES` 窗口内拒绝
- **设备哈希双读兼容**：新旧两种哈希都查，避免存量用户被误判为未注册而重复建号
- **客户端 IP 校验**：`X-Forwarded-For` / `X-Real-IP` 可伪造，非法值一律丢弃回落 `0.0.0.0`，防止脏数据触发存储型 XSS

## 6. 前端衔接（实际）

前端已对接后端设备认证，见 [frontend/src/lib/auth.ts](../frontend/src/lib/auth.ts)：

- 启动时生成并持久化 `X-Device-Id`（`getOrCreateDeviceId`）
- 调用 `/auth/device-status` 判断是否已注册
- 未注册则 `/auth/register-by-device`，已有账号则 `/auth/login`
- 令牌与账号缓存在本地，请求时通过 `Authorization: Bearer` 携带

> 前端已**只信任后端认证结果**，不存在独立的"本地假认证"账号体系。

## 8. 相关配置（backend/.env）

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `JWT_SECRET` | 无 | JWT 签发密钥，**必须设为强随机值** |
| `JWT_EXPIRATION_HOURS` | `720` | 令牌有效期（小时） |
| `PASSWORD_HASH_ITERATIONS` | `120000` | PBKDF2 迭代次数 |
| `LOGIN_FAILURE_LIMIT` | — | 登录失败阈值（用户名维度） |
| `LOGIN_FAILURE_WINDOW_MINUTES` | — | 失败计数窗口（分钟） |

> 配置加载见 [backend/src/config.rs](../backend/src/config.rs)。切勿将真实 `JWT_SECRET` 或数据库凭据提交到仓库。
