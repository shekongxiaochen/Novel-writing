# 更新手册

上线后改东西照这份做。先看你改了什么，对号入座：

| 你改了什么 | 看哪一章 | 麻烦度 | 用户要重装吗 |
|---|---|---|---|
| AI 提示词 / 密钥 / 价格 | 第一章 | ⭐ 后台点几下 | 不用 |
| 后端代码（新 API、改逻辑、改计费） | 第二章 | ⭐⭐ 重新部署后端 | 不用* |
| 前端界面（Vue 页面、交互） | 第三章 | ⭐⭐⭐ 重新打包分发 | 要重装 |

> *前提是 API 接口没变。若前后端一起改了且接口有变动，第二、三章都要做。

# 第一章：改 AI 提示词 / 密钥 / 价格（零部署，最常用）

这些东西存在数据库里，后台改完即时生效，**完全不用碰代码、不用碰服务器**。

## 步骤

1. 浏览器打开后台：`http://114.66.54.178:14675/admin/`
2. 用管理员账号登录（用户名 `admin`，密码在服务器 `.env.prod` 的 `ADMIN_PASSWORD`）
3. 改提示词：左侧菜单「AI 提示词」→ 找到要改的那条 → 点「编辑」→ 改内容 → 保存
   - 想还原成原始提示词：点「恢复默认」
4. 改 AI 密钥 / 模型 / 计费价格：左侧菜单「AI 配置」→ 改 → 保存

## 生效时间

立刻生效。用户下一次调用 AI 就用新的，无需重启、无需重装。

## 注意

- 改密钥后，先在客户端试一次 AI 功能，确认新密钥能用。
- 改计费价格要谨慎，影响所有用户的扣费。

# 第二章：改后端代码（重新部署后端）

改了 `backend/src/` 里的 Rust 代码（新增 API、改逻辑、改计费算法等），按这个流程。
分两大步：**本机打包上传** → **服务器解压并重新构建**。

> 重要：服务器目录 `/opt/novel-backend` 里，后端代码是**平铺在根目录**的
> （`src/`、`Cargo.toml`、`admin-templates/` 直接在根下）。
> 而本机仓库里后端在 `backend/` 子目录。所以上传后要把 `backend/` 里的内容
> 同步到 `/opt/novel-backend` 根目录，**不是**整个 backend 文件夹塞进去。

## A. 本机：打包并上传后端代码

在本机打开 Git Bash，进入项目根目录：

```bash
cd "d:/Novel writing"
```

打包后端（只传源码，排除编译产物，几百 KB）：

```bash
tar --exclude='backend/target' --exclude='backend/.env' --exclude='backend/dump.rdb' -czf backend-deploy.tar.gz backend
```

上传到服务器（会让你输服务器密码）：

```bash
scp -P 14672 backend-deploy.tar.gz root@114.66.54.178:/opt/
```

> `-P 14672` 是 SSH 端口（大写 P），不是 22。

## B. 服务器：解压并同步到正确目录

SSH 登录服务器：

```bash
ssh -p 14672 root@114.66.54.178
```

整段复制运行（解压后把内容同步到平铺的根目录）：

```bash
cd /opt
tar -xzf backend-deploy.tar.gz          # 解出 /opt/backend
# 同步到服务真正使用的目录（注意是 backend 里的内容 → novel-backend 根目录）
cp -rf /opt/backend/src/* /opt/novel-backend/src/
cp -rf /opt/backend/admin-templates/* /opt/novel-backend/admin-templates/
cp -f /opt/backend/Cargo.toml /opt/backend/Cargo.lock /opt/novel-backend/
cp -f /opt/backend/Dockerfile /opt/novel-backend/ 2>/dev/null || true
```

> ⚠️ 关键坑：tar 解压出的顶层目录叫 `/opt/backend`，但服务实际用的是
> `/opt/novel-backend`（带横线）。必须用上面的 cp 同步过去，否则更新不生效。
> `.env.prod` 和 `docker-compose.iponly.yml` 不在 tar 包里，**不会被覆盖**，密钥安全。

校验新代码到位（看修改时间是不是今天）：

```bash
ls -la --time-style=long-iso /opt/novel-backend/src/admin/
```

## C. 服务器：重新构建并重启

服务器上有现成的更新脚本 `update.sh`，直接跑：

```bash
cd /opt/novel-backend && bash update.sh
```

它会自动：检查文件 → 重新构建（Rust 增量编译，约 1-3 分钟）→ 重启 → 健康检查。
看到 `✅ 更新完成，后端已健康运行` 就好了。

> 脚本不加 `--no-cache`，保留依赖缓存避免国内网络重下超时。
> 只换后端程序，**绝不动数据库**，用户数据全部保留。

## D. 验证更新成功

```bash
docker compose -f docker-compose.iponly.yml --env-file .env.prod ps
curl -s http://localhost:8080/health; echo ""
```

- backend 显示 `Up ... (healthy)` 且 `/health` 返回正常 → 更新成功。
- 若 backend 反复 `Restarting`，看日志排查：
  `docker logs novel-backend --tail 30`

## 重要：数据库不会被动

重新构建**只换后端程序，不动数据库**。用户数据、小说、钱包全部保留。
（数据卷 mysql_data 一直在，除非你手动 `down -v` 才会清空——平时更新千万别加 `-v`。）

## 如果改了数据库结构（加了新表/新字段）

你在 `backend/migrations/` 加了新的 `.sql` 文件时，要注意：
`docker-entrypoint-initdb.d` 只在**数据库首次初始化**时执行，**已有数据库不会自动跑新迁移**。
这种情况别自己瞎试，让 Claude 帮你手动把新迁移 SQL 应用到现有数据库（避免误删数据）。

# 第三章：改前端界面（重新打包客户端）

改了 `frontend/src/` 里的 Vue 代码（界面、新页面、交互），要重新打包，
然后把新安装包发给用户，用户重新安装。

## A. 本机：重新打包

在 Git Bash 进入前端目录：

```bash
cd "d:/Novel writing/frontend"
```

打包（会自动用生产地址 `.env.production`，约 10-20 分钟）：

```bash
npm run tauri:build
```

> 不用手动改服务器地址。`.env.production` 已固定为 `http://114.66.54.178:14675`，
> 打包时自动读取。除非服务器地址变了才需要改它。

## B. 产物在哪

打包完成后，两个产物：

- **安装版**（推荐发给用户）：
  `frontend/src-tauri/target/release/bundle/nsis/Novel Writing Assistant_1.0.0_x64-setup.exe`
- **免安装版**（直接双击运行）：
  `frontend/src-tauri/target/release/novel-writing-assistant.exe`

## C. 发给用户 + 用户怎么更新

1. 把新的 `setup.exe` 发给用户（网盘、微信、你的官网下载链接等）。
2. 用户**直接安装新版**覆盖旧版即可，数据不会丢
   （用户的小说存在服务器，不在客户端本地）。

## D. 版本号管理（建议）

每次发新版前，改一下版本号，方便区分：
- `frontend/package.json` 里的 `"version"`
- `frontend/src-tauri/tauri.conf.json` 里的 `"version"`
- 两处改成一样，比如 `1.0.1`、`1.1.0`
改完版本号，产物文件名也会跟着变（如 `..._1.0.1_x64-setup.exe`）。

# 附录：常见问题

- **改完没生效**：提示词/密钥类在后台改的，刷新页面重试；代码类的确认有没有跑 `--build` 重新构建。
- **backend 起不来、反复重启**：`docker logs novel-backend --tail 30` 看报错，多半是数据库或配置问题。
- **客户端连不上**：先在浏览器开 `http://114.66.54.178:14675/health`，能返回 ok 说明后端没问题，问题在客户端或网络。
- **千万不要**在 `down` 命令后加 `-v`（会清空数据库），除非你确定要清空重来。
